# #7

## 문제 1

### 문제

회사는 단일 VPC의 Amazon EC2 인스턴스에서 고가용성 이미지 처리 애플리케이션을 실행합니다. EC2 인스턴스는 여러 가용 영역에 걸쳐 여러 서브넷 내에서 실행됩니다. EC2 인스턴스는 서로 통신하지 않습니다. 그러나 EC2 인스턴스는 Amazon S3에서 이미지를 다운로드하고 단일 NAT 게이트웨이를 통해 Amazon S3에 이미지를 업로드 합니다. 회사는 데이터 전송 요금에 대해 우려하고 있습니다.

회사가 지역 데이터 전송 요금을 피하는 가장 비용 효율적인 방법은 무엇입니까?

### 선택지

1. 각 가용 영역에서 NAT 게이트웨이를 시작합니다.
2. NAT 게이트웨이를 NAT 인스턴스로 교체합니다.
3. Amazon S3용 게이트 웨이 VPC 엔드포인트를 배포합니다.
4. EC2 인스턴스를 실행할 EC2 전용 호스트를 프로비저닝 합니다.

### 정답

3번

### 해설

문제의 핵심은 `EC2가 S3와 통신할 때 발생하는 지역 데이터 전송 요금`을 줄이는 것입니다.  
`Gateway VPC Endpoint for Amazon S3`를 사용하면 VPC 내부에서 사설 경로로 S3에 접근할 수 있으므로 NAT 게이트웨이를 거치지 않아도 되고, 추가적인 NAT 관련 데이터 전송 비용도 줄일 수 있습니다.

각 AZ마다 NAT 게이트웨이를 두는 것은 가용성에는 도움이 될 수 있지만 비용 절감의 직접적인 해법은 아닙니다. NAT 인스턴스로 교체하는 방식은 운영 부담이 커지고, 전용 호스트는 이 문제와 관련이 없습니다.

### 핵심 메모

- `Gateway VPC Endpoint`
  VPC에서 `S3`와 `DynamoDB`에 사설 경로로 접근할 수 있게 한다.
- `S3 Gateway Endpoint`
  NAT 게이트웨이를 거치지 않고 `S3`에 접근할 수 있어 비용 절감에 유리하다.
- `추가 요금`
  게이트웨이 VPC 엔드포인트 자체에는 추가 요금이 없다.

## 문제 2

### 문제

한 회사가 프로덕션 애플리케이션의 재해 복구(DR) 전략을 설계하고 있습니다. 애플리케이션은 us-east-1 지역의 Amazon Aurora 클러스터에 있는 MySQL 데이터베이스의 지원을 받습니다. 회사는 us-west-1 지역을 DR 지역으로 선택했습니다.

회사의 목표 복구 지점 목표(RPO)는 5분이고 목표 복구 시간 목표(RTO)는 20분입니다. 회사에서는 구성 변경을 최소화하려고 합니다.

가장 효율적인 운영 효율성으로 이러한 요구 사항을 충족하는 솔루션은 무엇입니까?

### 선택지

1. 프로덕션 애플리케이션의 Aurora MySQL 클러스터 라이터 인스턴스와 비슷한 크기로 us-west-1에 Aurora 읽기 전용본을 생성합니다.
2. Aurora 클러스터를 Aurora 글로벌 데이터베이스로 변환하십시오. 관리형 장애 조치를 구성합니다.
3. 교차 리전 복제 기능이 있는 us-west-1에 새 Aurora 클러스터를 생성합니다.
4. us-west-1에 새 Aurora 클러스터를 생성합니다. AWS Database Migration Service(AWS DMS)를 사용하여 두 클러스터를 모두 동기화 합니다.

### 정답

2번

### 해설

요구 사항은 `RPO 5분`, `RTO 20분`, `구성 변경 최소화`, 그리고 `운영 효율성`입니다.  
`Amazon Aurora Global Database`는 리전 간 저지연 복제를 제공하며, 재해 발생 시 DR 리전으로 빠르게 장애 조치할 수 있어 엄격한 RPO/RTO 요구 사항을 만족하기 좋습니다.

또한 Aurora의 기본 기능을 활용하므로 별도 동기화 도구를 운영하는 것보다 관리 부담이 적습니다. `DMS`를 사용한 동기화나 수동 교차 리전 복제 구성은 더 복잡하고 운영 오버헤드가 큽니다.

### 핵심 메모

- `Aurora Global Database`
  여러 리전에 걸쳐 저지연 복제를 제공해 DR 전략에 적합하다.
- `RPO/RTO`
  엄격한 복구 요구 사항에서는 관리형 교차 리전 복제 기능이 유리하다.
- `운영 효율성`
  별도 복제 솔루션보다 Aurora 기본 기능 활용이 더 단순하다.

## 문제 3

### 문제

회사는 ALB(Application Load Balancer) 뒤에 있는 Amazon EC2 인스턴스에서 웹 사이트를 호스팅합니다. 웹사이트는 정적 콘텐츠를 제공합니다. 웹사이트 트래픽이 증가하고 있습니다. 회사는 웹사이트 호스팅 비용을 최소화하려고 합니다.

이러한 요구 사항을 충족하는 솔루션은 무엇입니까?

### 선택지

1. 웹 사이트를 Amazon S3 버킷으로 이동합니다. S3 버킷에 대한 Amazon CloudFront 배포를 구성합니다.
2. 웹 사이트를 Amazon S3 버킷으로 이동합니다. S3 버킷에 대한 Amazon ElastiCache 클러스터를 구성합니다.
3. 웹 사이트를 AWS Amplify로 이동합니다. Amplify 웹 사이트를 확인하도록 ALB를 구성합니다.
4. 웹 사이트를 AWS Amplify로 이동합니다. 웹 사이트를 캐시하도록 EC2 인스틴스를 구성합니다.

### 정답

1번

### 해설

웹사이트가 `정적 콘텐츠`만 제공한다면 `EC2 + ALB` 조합은 과한 구성이며 비용도 더 많이 듭니다.  
정적 사이트는 `Amazon S3`에 저장하고, 전 세계 캐싱과 성능 최적화를 위해 `Amazon CloudFront`를 앞단에 두는 것이 가장 일반적이고 비용 효율적인 구조입니다.

`ElastiCache`는 주로 데이터베이스나 애플리케이션 응답 캐시에 사용되며 S3 정적 사이트 앞단 구성과는 맞지 않습니다. `Amplify`도 정적 웹 배포에 사용할 수는 있지만, 보기 1이 더 직접적이고 AWS 시험 관점에서도 표준 해법입니다.

### 핵심 메모

- `Amazon S3`
  정적 웹사이트 호스팅에 적합하고 비용 효율적이다.
- `Amazon CloudFront`
  엣지 캐싱으로 성능을 높이고 원본 요청을 줄일 수 있다.
- `정적 콘텐츠`
  서버 기반 EC2 호스팅보다 `S3 + CloudFront` 조합이 일반적으로 더 저렴하다.

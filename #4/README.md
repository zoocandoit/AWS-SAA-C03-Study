# #4

## 문제 1

### 문제

회사는 Amazon DynamoDB를 사용하여 고객 정보를 저장하는 쇼핑 애플리케이션을 실행합니다.

데이터 손상의 경우 솔루션 설계자는 15분의 RPO(복구 지점 목표)와 1시간의 RTO(복구 시간 목표)를 충족하는 솔루션을 설계해야 합니다.

이러한 요구 사항을 충족하기 위해 솔루션 설계자는 무엇을 권장해야 합니까?

### 선택지

1. DynamoDB 전역 테이블을 구성합니다. RPO 복구의 경우 애플리케이션이 다른 AWS 리전을 가리키도록 합니다.
2. DynamoDB 특정 시점으로 복구를 구성합니다. RPO 복구를 위해 원하는 시점으로 복원합니다.
3. 매일 DynamoDB 데이터를 Amazon S3 Glacier로 내보냅니다. RPO 복구를 위해 S3 Glacier에서 DynamoDB로 데이터를 가져옵니다.
4. 15분마다 DynamoDB Amazon Elastic Block Store(Amazon EBS) 스냅샷을 예약합니다. RPO 복구의 경우 EBS 스냅샷을 사용하여 DynamoDB 테이블을 복원합니다.

### 정답

2번

### 해설

핵심 요구 사항은 `데이터 손상`에 대비하면서 `RPO 15분`, `RTO 1시간`을 만족하는 것입니다.  
`DynamoDB Point-in-Time Recovery(PITR)`를 사용하면 최근 35일 이내 임의의 시점으로 테이블을 복구할 수 있어, 데이터 손상 직전 시점으로 되돌리는 데 적합합니다.

`Global Tables`는 리전 장애나 고가용성에는 유용하지만, 데이터 손상이 발생하면 손상된 데이터가 다른 리전에도 복제될 수 있어 이 문제의 핵심 해결책이 아닙니다. `S3 Glacier` 일일 내보내기는 RPO 15분을 충족하지 못하고, `DynamoDB`는 `EBS` 스냅샷 기반으로 복구하지 않습니다.

### 핵심 메모

- `DynamoDB PITR`
  최근 35일 내 특정 시점으로 테이블을 복구할 수 있다.
- `Global Tables`
  리전 장애 대응에는 강하지만 논리적 데이터 손상 복구의 정답은 아니다.
- `RPO 15분`
  일일 백업으로는 충족할 수 없다.

## 문제 2

### 문제

회사에는 중요한 데이터가 포함된 Amazon S3 버킷이 있습니다.

회사는 우발적인 삭제로부터 데이터를 보호해야 합니다.

이러한 요구 사항을 충족하기 위해 솔루션 설계자는 어떤 단계 조합을 수행해야 합니까? (두 가지를 선택하세요.)

### 선택지

1. S3 버킷에서 버전 관리를 활성화합니다.
2. S3 버킷에서 MFA 삭제를 활성화합니다.
3. S3 버킷에 버킷 정책을 생성합니다.
4. S3 버킷에서 기본 암호화를 활성화합니다.
5. S3 버킷의 객체에 대한 수명 주기 정책을 생성합니다.

### 정답

1번, 2번

### 해설

우발적인 삭제 방지의 핵심은 `삭제되더라도 복구 가능해야 함`과 `삭제 자체를 더 어렵게 만들어야 함`입니다.  
`Versioning`을 활성화하면 객체가 삭제되거나 덮어써져도 이전 버전을 보존할 수 있고, `MFA Delete`를 사용하면 버전 삭제나 버전 관리 변경 시 추가 인증을 요구해 실수로 삭제하는 위험을 줄일 수 있습니다.

버킷 정책은 접근 제어에는 유용하지만 우발적 삭제 방지의 핵심 기능은 아니고, 기본 암호화는 데이터 보호 목적이 다릅니다. 수명 주기 정책은 오히려 객체 삭제를 자동화할 수 있어 이 요구 사항과는 맞지 않습니다.

### 핵심 메모

- `S3 Versioning`
  삭제나 덮어쓰기가 발생해도 이전 버전을 복원할 수 있다.
- `S3 MFA Delete`
  객체 버전 삭제나 버전 관리 변경에 MFA를 요구한다.
- `기본 암호화`
  삭제 방지와는 별개의 기능이다.

## 문제 3

### 문제

회사는 Amazon S3에 데이터를 저장하고 데이터가 변경되지 않도록 해야 합니다.

회사는 Amazon S3에 업로드된 새 객체가 회사에서 객체를 수정하기로 결정할 때까지 불특정한 시간 동안 변경 불가능한 상태로 유지되기를 원합니다.

회사 AWS 계정의 특정 사용자만 개체를 삭제할 수 있습니다.

솔루션 설계자는 이러한 요구 사항을 충족하기 위해 무엇을 해야 합니까?

### 선택지

1. S3 Glacier 볼트를 생성합니다. 개체에 WORM(Write-Once, Read-Many) 볼트 잠금 정책을 적용합니다.
2. S3 객체 잠금이 활성화된 S3 버킷을 생성합니다. 버전 관리를 활성화합니다. 보존 기간을 100년으로 설정합니다. 거버넌스 모드를 새 객체에 대한 S3 버킷의 기본 보관 모드로 사용합니다.
3. S3 버킷을 생성합니다. AWS CloudTrail을 사용하여 객체를 수정하는 모든 S3 API 이벤트를 추적합니다. 알림을 받으면 회사가 가지고 있는 모든 백업 버전에서 수정된 개체를 복원합니다.
4. S3 객체 잠금이 활성화된 S3 버킷을 생성합니다. 버전 관리를 활성화합니다. 객체에 법적 보존을 추가합니다. 객체를 삭제해야 하는 사용자의 IAM 정책에 s3:PutObjectLegalHold 권한을 추가합니다.

### 정답

4번

### 해설

문제의 핵심은 `불특정한 시간 동안 변경 불가`, 그리고 `특정 사용자만 삭제 가능`입니다.  
`S3 Object Lock`의 `Legal Hold`는 종료 날짜 없이 객체를 변경 불가 상태로 유지할 수 있으므로, "회사가 수정하기로 결정할 때까지"라는 요구 사항에 잘 맞습니다.

또한 특정 사용자에게만 `s3:PutObjectLegalHold` 권한을 부여하면, 해당 사용자만 법적 보존을 해제한 뒤 삭제 작업을 진행할 수 있습니다.  
2번의 `100년 보존 기간`은 사실상 장기 보관이지만 "불특정한 시간"과는 다르고, 보존 기간이 끝나기 전까지 유연하게 해제하는 요구와도 정확히 일치하지 않습니다.

### 핵심 메모

- `S3 Object Lock`
  객체를 WORM 형태로 보호할 수 있다.
- `Legal Hold`
  만료일 없이 객체를 보호할 수 있어 기간이 정해지지 않은 보존 요구에 적합하다.
- `Governance Mode`
  보존 기간 기반 제어에는 유용하지만, 이 문제는 `Legal Hold`가 더 정확히 맞다.
